# todomcp (jhirono)

## What It Is

The **original** Microsoft To Do MCP server by Jumpei Hirono (14 commits, sole maintainer, last commit March 31, 2025). Jordan Burke's `microsoft-todo-mcp-server` (audited separately) is a fork of this repo. Written in TypeScript + JavaScript, published on npm as `@jhirono/todomcp`. Provides 13 MCP tools for CRUD operations on task lists, tasks, and checklist items, plus an auth-status tool.

## How It Works

- **MCP Server** (`todo-index.ts`, ~1500 lines): Registers 13 tools with `@modelcontextprotocol/sdk`, uses stdio transport. All API calls go through a single `makeGraphRequest()` helper targeting `https://graph.microsoft.com/v1.0`.
- **Auth Server** (`auth-server.js`, ~500 lines): Express server on `localhost:3000` for OAuth 2.0 authorization code flow using `@azure/msal-node` v1.x `ConfidentialClientApplication`. Runs only during initial setup. Includes `/refresh` and `/silentLogin` endpoints.
- **CLI Entry** (`cli.ts`, ~50 lines): Reads tokens from env vars or `tokens.json`, starts the MCP server.
- **Config Generator** (`create-mcp-config.ts`, ~50 lines): Reads `tokens.json` and produces an `mcp.json` file with tokens embedded as env vars for Cursor/Claude integration.

Key architectural difference from the fork: **No setup wizard, no auto-modification of Claude Desktop config**. Uses a simpler `mcp.json` export approach instead.

### Hardcoded Tenant Bug

The `refreshAccessToken()` function in `todo-index.ts` hardcodes the token endpoint to:
```
https://login.microsoftonline.com/consumers/oauth2/v2.0/token
```

This ignores the `TENANT_ID` env var, meaning token refresh will **break for organizational accounts** that use a different tenant. The auth-server.js correctly reads `TENANT_ID`, but the MCP server's built-in refresh does not.

### Data Exfiltration Risk

**All network endpoints contacted (exhaustive list)**:

1. `https://graph.microsoft.com/v1.0/me/todo/lists` вЂ” Microsoft Graph API (tasks)
2. `https://graph.microsoft.com/v1.0/me` вЂ” User info (personal account detection)
3. `https://login.microsoftonline.com/{tenantId}/oauth2/v2.0/token` вЂ” Auth server token endpoint
4. `https://login.microsoftonline.com/consumers/oauth2/v2.0/token` вЂ” MCP server token refresh (hardcoded)
5. `https://graph.microsoft.com/.default` вЂ” Client credentials scope (via `/silentLogin` endpoint)
6. `http://localhost:3000/callback` вЂ” Local auth callback

**No telemetry, no analytics, no phone-home, no third-party endpoints.** All traffic goes exclusively to Microsoft's official endpoints or localhost. Verified by full grep of all source files.

### `/silentLogin` Endpoint

The auth-server exposes a `/silentLogin` endpoint that uses `acquireTokenByClientCredential` with `https://graph.microsoft.com/.default` scope. This is a **client credentials flow** (application-only, no user context) вЂ” it would grant full app-level permissions if an admin has consented. While not automatically dangerous (requires Azure admin consent), it's an unnecessary attack surface for a tool that only needs delegated user permissions.

## Red Flags

1. **No LICENSE file** вЂ” legally unclear for adoption or forking
2. **13 months since last commit** вЂ” effectively abandoned
3. **piiLoggingEnabled: true** вЂ” logs PII to stdout during auth
4. **Hardcoded `consumers` tenant in token refresh** вЂ” breaks organizational accounts
5. **82 direct dependencies** вЂ” all transitive deps pinned as direct (sloppy, makes auditing harder)
6. **Dead dependencies** вЂ” `express-rate-limit` and `pkce-challenge` declared but never imported or used
7. **MSAL v1 on Node 24** вЂ” engine incompatibility, v1 is end-of-life
8. **MCP SDK v1.7** вЂ” significantly behind current v1.21+
9. **`/silentLogin` endpoint** вЂ” exposes client credentials flow with `.default` scope, unnecessary attack surface
10. **No tests** вЂ” zero automated test coverage
11. **Auth callback shows partial tokens** вЂ” truncated tokens visible in browser HTML
12. **Solo maintainer, no succession** вЂ” single developer, no org backing

## What todomcp Does Better Than jordanburke

1. **Doesn't auto-modify external application configs** вЂ” exports `mcp.json` instead of writing directly into Claude Desktop's config
2. **Doesn't store client secret in `tokens.json`** вЂ” client secret stays in `.env` only
3. **No debug tools in production** вЂ” no `test-graph-api-exploration` shipped
4. **Personal account detection** вЂ” warns proactively about MailboxNotEnabledForRESTAPI before the user hits the error
5. **OData query support** вЂ” `get-tasks` tool exposes `filter`, `select`, `orderby`, `top`, `skip`, `count` parameters (jordanburke has similar but less documented)
6. **Tool descriptions are clearer** вЂ” each tool has a well-written description explaining the Microsoft To Do hierarchy

## What todomcp Does Worse Than jordanburke

1. **MSAL v1 vs v3** вЂ” outdated auth library with Node compatibility issues
2. **MCP SDK v1.7 vs v1.21** вЂ” older protocol version
3. **No LICENSE file** вЂ” can't legally fork or redistribute
4. **13 months stale vs 5 months** вЂ” less maintained
5. **82 spurious direct dependencies** вЂ” sloppy package.json
6. **Hardcoded tenant in refresh** вЂ” breaks multi-tenant scenarios
7. **Auth server is JavaScript, not TypeScript** вЂ” inconsistent codebase
8. **`/silentLogin` endpoint** вЂ” unnecessary client credentials flow exposure
9. **No organized list view** вЂ” lacks jordanburke's `get-task-lists-organized`
10. **No archive/bulk operations** вЂ” lacks jordanburke's `archive-completed-tasks`

## What We Should Adopt From This Implementation

1. **Personal account detection pattern** вЂ” the `isPersonalMicrosoftAccount()` helper that checks email domain and warns proactively is good UX. Adopt: detect and warn at startup, not on first error.
2. **OData query parameter exposure** вЂ” the `get-tasks` tool's full OData support (`filter`, `select`, `orderby`, `top`, `skip`, `count`) is well-designed. Our spec should match this.
3. **Tool description quality** вЂ” every tool description explains the Microsoft To Do hierarchy and what the entity represents. Copy this pattern for our tool descriptions.
4. **No external config modification** вЂ” confirm our approach of never auto-modifying Claude/Cursor config files. Let the user copy the config themselves.
5. **Empty-string-to-null pattern** вЂ” `update-task` supports empty string for date fields to clear them (sets to `null`). Useful for AI agents that need to remove due dates.
